C++协程：理解 co_await 操作

在上一篇文章中，我叙述了协程和函数之间的高级差异，但是没有详细介绍 C++ Coroutine TS（N4680）中所描述的语法和语义。

Coroutines TS 添加到C++中的关键功能是暂停一个协程，并在之后将其恢复执行的能力。TS 为此所提供的机制是通过新操作 co_await 来完成的。

理解 co_await 操作时如何工作的有助于我们揭开协程行为的神秘面纱，以及协程是如何暂停和恢复的。在这篇文章中我会解释 co_await 操作的机制，并介绍相关的 Awaitable 和 Awaiter 类型概念。

但是在深入研究 co_await 之前，我想要先简述一下 Coroutines TS ，以供读者理解其来龙去脉。

Coroutines TS 给了我们什么

  • 三个新的语言关键字： co_await , co_yield 和 co_return
  • 在 std::experimental 命名空间下的一些新的类型（现在 C++20 编译器可以直接使用 std 命名空间了——译注）：
    • coroutine_handle<P>
    • coroutine_traits<Ts…>
    • suspend_always
    • suspend_never
  • 一个库作者可以与协程交互并自定义其行为的通用机制
  • 一个语言设施使得编写异步代码更加容易

C++ Coroutines TS 所提供的这些语言设施可以认为是协程的低级汇编语言。其难以直接安全使用，主要是为了库作者使用，使他们可以构建让应用开发人员可以安全使用的更高级别的抽象。

计划是在即将到来的语言标准中交付这些新的语言设施（希望是C++20）（已经在C++20中添加了——译注），以及标准库中附带的一些高级类型，这些高级类型包装这些低级构建模块，使协程可以供应用程序开发者以一种安全方式使用。

编译器 <-> 库 的交互

有趣的是，Coroutines TS 事实上没有定义协程的语义。它既没有定义如何生成给调用方的返回值，也没有定义如何传递返回值到 co_return 语句，又没有定义如何从协程中抛出异常，还没有定义协程应该从哪个线程恢复。

相反，它为库代码指定了一种通用机制，通过实现特定接口的类型来定制协程的行为。之后编译器会生成代码，在库提供的类型实例中调用方法。正如库作者可以定义 begin() / end() 方法和 iterator 类型来定制基于范围的循环那样。

Coroutines TS 并没有为协程机制规定任何语义，这一事实使其成为一个强大的工具。其允许库作者可以为不同的目的定义许多不同种类的协程。

例如，你可以定义一个异步提供单个值的协程，或者延迟生成一系列值的协程，还可以是简化控制流程的协程，比如在 optional<T> 中遇到 nullopt 就提前结束对其中值的使用。

Coroutines TS 定义了两种类型的接口：Promise 接口和 Awaitable 接口。

Promise 接口指定用于自定义协程自身行为的方法。库作者可以通过其定义协程调用的时候会发生什么，协程返回的时候会发生什么（包括正常方式或未经处理的异常），以及自定义协程中任何 co_await 和 co_yield 表达式的行为。

Awaitable 接口所指定的方法则可以控制 co_await 表达式的语义。当一个值接在 co_await 表达式之后时，代码将转换成对 awaitable 对象上一些列方法的调用，这些调用可以指定：是否暂停当前协程；协程挂起以后执行一些逻辑以计划协程之后的恢复；协程恢复后执行一些逻辑以向 co_await 表达式提供结果。

我会在以后的文章中详细叙述 Promise 接口，但是现在我们先来看一下 Awaitable 接口。

Awaiters 和 Awaitables ：解释 co_await 运算符

co_await 运算符是一个新的一元运算符，可以适用于值。例如：co_await someValue。

co_await 运算符只能在协程上下文中应用。不过，这在某种程度上属于冗余表达，因为根据定义，任何函数体包含 co_await 运算符都会被编译为协程。

我们称支持 co_await 运算符的类型为 Awaitable 类型。

注意 co_await 运算符是否可以用于某个类型取决于 co_await 所在表达式的上下文。协程使用的承诺类型（promise type）可以通过 await_transform 方法来更改一个 co_await 表达式的含义（之后会详细介绍）。

为了能更具体地说明，我想用术语 Normally Awaitable 来描述在协程上下文中支持 co_await 运算符的类型，并且这个类型的承诺类型中并没有 await_transform 成员。我还要用术语 Contextually Awaitable 来描述一种在某些类型的协程上下文中仅支持 co_await 运算符的类型，这种类型的协程承诺类型中存在 await_transform 成员。（作者在这等着大家给他提供关于这些术语名字更好的建议。）

一个 Awaiter 类型会实现三个特殊方法，它们会被作为 co_await 的一部分调用，它们分别是：await_ready、await_suspend 以及 await_resume。

请注意，我在这里稍微从 C# 的 async 关键字机制中“窃”了术语“Awaiter”，该机制依据了 GetAwaiter() 方法，此方法中返回的对象的接口与 C++ 中 Awaiter 的概念惊人地相似。在这篇文章中可以找到关于 C# awaiter 的更多信息。

注意一个类型可以同时是 Awaitable 类型和 Awaiter 类型。

当编译器看到一个 co_await <expr> 表达式时，实际上可以根据其所涉及的类型，来将其转换为许多可能的东西。

获取 Awaiter

编译器所做的第一件事是为获取等待值的 Awaiter 对象来生成代码。在 N4680 的 5.3.8(3) 节中，陈述了获取 awaiter 对象的许多步骤。

让我们假设等待中的协程承诺对象（promise object）具有类型 P，并且 promise 是当前协程承诺对象的一个左值引用。

如果这个承诺类型 P 有一个叫 await_transform 的成员，那么 <expr> 会首先传递到 promise.await_transform(<expr>) 中以获取 Awaitable 的值 awaitable。否则，如果承诺类型没有 await_transform 成员，那么我们就直接使用计算 <expr> 的结果来作为 awaitable。

当调用的 await_suspend() 返回时，void 返回版本的 await_suspend 会无条件地将执行转移回协程的调用者/恢复器。与此不同的是，bool 返回的版本可以允许 awaiter 对象有条件地立即恢复协程而不需要返回到调用者/恢复器。

bool 返回版本的 await_suspend() 会在这种情况下很有用：awaiter 启动了一个有时可以同步完成的异步操作。在同步完成的情况下，await_suspend() 方法就可以返回 false 来指示协程能立即恢复并继续执行。

在 <suspend-coroutine> 这个地方编译器会生成一些代码来保存协程的当前状态，并为其恢复做准备。这包括存储 resume-point 的位置，以及将寄存器持有的任意值保存到协程帧的内存中。

在 <suspend-coroutine> 操作完成后，就可以认为当前协程挂起了。这个挂起的协程中你可以观察到的第一个断点在对 await_suspend() 的调用中。一旦协程挂起了，之后就可以对其进行恢复或销毁操作。

await_suspend() 方法的责任是在未来的某个点上，一旦操作完成，那么就调度协程恢复（或者销毁）。注意从 await_suspend() 中返回 false 视为调度协程在当前线程上立即恢复。

await_ready() 方法的目的是允许你在操作会同步完成，不需要挂起的情况下，去避免 <suspend-coroutine> 所带来的消耗。

在 <return-to-caller-or-resumer> 断点处执行会转移回调用方或恢复器，将局部栈帧弹出，但是保持协程帧的生命周期不被销毁。

当挂起的协程最终恢复时，执行会恢复到 <resume-point> 这个点上，即直接在 await_resume() 方法调用之前获取到操作的结果。

对 await_resume() 方法调用的返回值会变成 co_await 表达式的结果。await_resume() 方法也可以抛出异常，在这种情况下异常会从 co_await 表达式中传出。

注意到如果异常从 await_suspend() 的调用中传出，那么协程会立即恢复，并且异常会从 co_await 表达式里传出且不会调用 await_resume()。

协程句柄（Coroutine Handles）

你可能已经注意到了对 coroutine_handle<P> 类型的使用，这个类型会传递到 co_await 表达式里 await_suspend() 的调用中。

此类型代表了一个协程帧的非所有权句柄，可用于恢复协程与销毁协程帧。其还可用于获取承诺对象。

当实现 Awaitable 类型时，你将在 coroutine_handle 上使用的关键方法为 .resume()，该方法应在操作完成后调用，以便将等待中的协程恢复执行。在 coroutine_handle 中调用 .resume() 会重新在 resume-point 出重新激活已挂起的协程。调用 .resume() 将在协程下次命中 <return-to-caller-or-resumer> 断点处返回。

.destroy() 方法会销毁协程帧，调用作用域内变量的析构函数并释放协程帧所使用的内存。你通常不需要（或者说实际上应当尽量避免）调用 .destroy() 方法，除非你是实现协程承诺类型的库编写者。通常情况下，协程帧将会为某些类型的从协程返回的 RAII 对象所拥有。所以在没有与 RAII 对象配合的情况下调用 .destroy() 会导致双重析构的 bug。

.promise() 方法返回一个协程承诺对象的引用。但是，正如 .destroy() 那样，其通常仅在编写协程承诺类型时有作用。你应该将协程的承诺类型视为协程内部的实现细节。对于大多数 Normally Awaitable 类型，你应该将 await_suspend() 方法的参数设置为 coroutine_handle<void> 而不是 coroutine_handle<Promise>。

coroutine_handle<P>::from_promise(P& promise) 函数允许从一个协程承诺对象的引用重新创建协程句柄。需要注意的是，你必须确保类型 P 与协程帧实际的承诺类型完全匹配。如果实际承诺类型是 Derived 而试图创建 coroutine_handle<Base> 则会导致未定义行为。

.address() 与 from_address() 函数允许将一个协程句柄转换为/转换自一个 void* 指针。这主要是为了将“上下文”参数传递到已存在的C风格的 API 中，所以你会发现这在某些情况下实现 Awaitable 类型时非常有用。然而在大多数情况下，我发现有必要在这个“上下文”参数中传递一些额外的信息到回调中，因此我通常最终在一个结构体中存储 coroutine_handle ，并在“上下文”参数中传递此结构体的指针，而不是使用 .address() 的返回值。

无需同步的异步代码

co_await 的一个最强大的设计特性，是其在协程挂起之后，执行返回给调用者/恢复器之前，拥有执行代码的能力。

这使得 Awaiter 对象可以在协程已经挂起的情况下启动一个异步操作，并将这个挂起协程的 coroutine_handle 传递给这个操作。当操作完成时（可能是在另一个线程），协程可以安全地恢复并不需要任何额外的同步操作。

例如，当协程已经挂起时，在 await_suspend() 中启动一个异步读取操作，意味着我们可以在操作完成时，直接恢复协程，而不需要任何线程同步的操作来协调开始操作的线程和完成操作的线程。

有一点需要 特别 注意，就是在使用这个方法时，一旦你将协程句柄发布到其它线程，那么另一个线程可能会在 await_suspend() 返回之前恢复协程，并可能继续并发地执行 await_suspend() 方法的剩余部分。

协程恢复执行时，所做的第一件事是调用 await_resume() 来获取结果，然后通常会立即销毁 Awaiter 对象（即 await_suspend() 调用的 this 指针）。接着协程可能会在 await_suspend() 返回之前执行完毕，销毁协程和承诺对象。

所以在 await_suspend() 方法中，一旦可能在另一个线程上并发地恢复该协程，你需要确保避免访问 this 或者协程的 .promise() 对象，因为这两者都可能已被销毁。通常，在操作开始后并调度协程恢复执行时，只有 await_suspend() 中的局部变量才可以安全访问。

与有栈协程的比较

我想要快速将 Coroutine TS 中的无栈协程与现在常见的一些有栈协程进行比较，例如 Win32 纤程（Win32 fibers）或 boost::context，来比较它们在协程挂起之后执行一些逻辑的能力。

在许多有栈协程的框架中，一个协程的挂起操作与另一个协程的恢复操作绑定到一个“上下文切换（context-switch）”操作中。在这个“上下文切换”操作中，通常没有机会在一个协程挂起、另一个协程运行之前，执行一些逻辑。

这就意味着，如果我们想要在有栈协程中实现相似的异步文件读取操作，我们需要在挂起协程 之前 启动这个操作。因此，可能在协程挂起并有条件恢复之前，操作在其它线程完成。这种操作完成与其它线程与协程挂起之间的潜在竞争，需要一些线程同步机制来仲裁。

一个可能的办法是使用蹦床上下文（trampoline context）来解决这个问题，其可在初始上下文挂起之后，代表这个初始上下文执行一个操作。但是这需要一些额外的基础设施和额外的上下文切换来达成，而这样做所带来的开销可能会大于试图避免的同步操作带来的成本。

避免内存分配

异步操作经常需要存储每个操作的状态，以保持跟踪操作的进度。这种状态通常会持续到操作结束，只有在操作完成时才会释放。

例如，调用异步的 Win32 I/O 函数需要你分配一个指向 OVERLAPPED 结构的指针。调用者有责任确保该指针在操作完成之前一直有效。

在传统的基于回调的 API 中，这个状态通常需要分配在堆上，来保证其有合适的生命周期。如果你执行很多操作，你就需要为这些操作一一分配和释放这个状态。如果性能出现问题则需要从一个自定义的内存分配池中来分配这些状态对象。

但是，当我们使用协程，我们可以避免存储从堆上分配的操作状态，而是利用局部变量都存储在协程帧中这一事实，来确保协程挂起时这些变量依旧有效。

通过将每次操作的状态放入 Awaiter 对象中，我们可以有效地从协程帧中“借用”内存，来在 co_await 表达式持续期间存储各个各个操作的状态。一旦操作完成，协程恢复，Awaiter 对象销毁，释放协程帧内存以供其它局部变量使用。

最终，协程帧可能依旧会分配在堆上。但是，一旦分配，这个协程帧可以用于许多异步操作，而只需在堆上分配一次。

仔细想想，协程帧充当了一种十分高性能的 arena 内存分配器角色。编译器在编译期计算出所有局部变量所需内存的总大小，并根据需要将这些内存分配给局部变量，而且没有额外开销！试着实现一个自定义分配器来打败它吧😁

例：实现一个简单的线程同步原语

一旦完成上述操作后，我们需要尝试将 awaiter 原子地插入到等待者的链表中。如果我们成功插入则返回 true 表示我们不想立即恢复协程，否则如果我们发现已被并发地改变为“已设置”状态，那么我们就返回 false 来表明需要立即恢复协程。

注意，我们加载旧状态时使用 “acquire” 内存序，故如果我们读取到特殊的 “set” 值，我们就可以看到在调用 “set()” 之前的写入。

如果 compare-exchange 调用成功我们需要 “release” 语义，这样后续 “set()” 的调用就会看到我们对 m_awaitingCoroutine 的写入以及之前对协程状态的写入。

填充事件类的剩余部分

现在我们定义了 awaiter 类型，让我们回头来看一下 async_manual_reset_event 中方法的实现。

首先，构造函数。如果是“未设置”状态那么需要初始化链表为空（即 nullptr），否则就初始化为“已设置”状态（即 this）。

接着，is_set() 方法就很直截了当——如果它有特殊值 this 那么就是“已设置”状态。

这之后是 reset() 方法。如果是“已设置”状态我们需要将其变回空列表的“未设置”状态，否则就将其保持原貌。

对于 set() 方法，我们希望通过一个特殊的值 this 来改变当前状态到“已设置”状态，然后检查旧值。如果存在正在等待的协程，那么我们会依次恢复它们，之后再返回。

最后，我们需要实现 operator co_await() 方法。这只需要构造一个 awaiter 对象。

如此我们就完成了。一个可等待异步手动复位的事件，无锁，无需内存分配，noexcept 的实现。
